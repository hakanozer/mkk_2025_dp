SOLID Prensipleri

Single Responsibility Principle (SRP)

Tanım: Bir sınıfın yalnızca bir sorumluluğu olmalı ve değişmesi için yalnızca bir nedeni bulunmalıdır.

Neden-Sonuç İlişkisi:
  • Neden: Kod değişikliklerinin yan etkilerinin minimize edilmesi
  • Sonuç: Daha az bağımlılık, daha kolay test edilebilirlik ve bakım
Kötü Kullanım:
@Service
public class UserService {
  // Kullanıcı işlemleri
  public User createUser(User user) { ... }
  public User updateUser(User user) { ... }

  // E-posta işlemleri (SRP ihlali!)
  public void sendEmail(User user, String message) { ... }

  // Loglama işlemleri (SRP ihlali!)
  public void logUserAction(User user, String action) { ... }
}
İyi Kullanım:
@Service
public class UserService {
  @Autowired
  private EmailService emailService;
  @Autowired
  private LoggingService loggingService;

  public User createUser(User user) {
    // Kullanıcı oluşturma mantığı
    loggingService.logUserCreation(user);
    emailService.sendWelcomeEmail(user);
    return user;
  }
}
Open Closed Principle (OCP)

Tanım: Yazılım varlıkları (sınıflar, modüller, fonksiyonlar) genişletmeye açık, değişikliğe kapalı olmalıdır.

Neden-Sonuç İlişkisi:
  • Neden: Mevcut kodu değiştirmeden yeni özellikler ekleyebilme
  • Sonuç: Daha az regresyon riski ve daha istikrarlı kod tabanı
Kötü Kullanım:
public class DiscountService {
  public double calculateDiscount(Order order, String customerType) {
    if ("REGULAR".equals(customerType)) {
      return order.getTotal() * 0.1;
    } else if ("VIP".equals(customerType)) {
      return order.getTotal() * 0.2;
    } else if ("PREMIUM".equals(customerType)) {
      return order.getTotal() * 0.3;
    }
    // Yeni müşteri tipi eklemek için kodu değiştirmek gerekir
    return 0;
  }
}
İyi Kullanım:
public interface DiscountStrategy {
  double calculateDiscount(Order order);
}

@Component
public class RegularDiscount implements DiscountStrategy {
  public double calculateDiscount(Order order) {
    return order.getTotal() * 0.1;
  }
}

@Service
public class DiscountService {
  @Autowired
  private Map<String, DiscountStrategy> strategies;

  public double calculateDiscount(Order order, String customerType) {
    DiscountStrategy strategy = strategies.get(customerType);
    if (strategy == null) {
      return 0;
    }
    return strategy.calculateDiscount(order);
  }
}
Liskov Substitution Principle (LSP)

Tanım: Alt sınıflar, üst sınıfların yerine geçebilmeli ve programın doğruluğunu bozmamalıdır.

Neden-Sonuç İlişkisi:
  • Neden: Kalıtım hiyerarşilerinde beklenmeyen davranışların önlenmesi
  • Sonuç: Daha güvenilir ve tahmin edilebilir kod
Kötü Kullanım:
class Rectangle {
  protected int width;
  protected int height;

  public void setWidth(int width) { this.width = width; }
  public void setHeight(int height) { this.height = height; }
}

class Square extends Rectangle {
  @Override
  public void setWidth(int width) {
    super.setWidth(width);
    super.setHeight(width); // LSP ihlali!
  }

  @Override
  public void setHeight(int height) {
    super.setHeight(height);
    super.setWidth(height); // LSP ihlali!
  }
}
İyi Kullanım:
interface Shape {
  int getArea();
}

class Rectangle implements Shape {
  private int width;
  private int height;

  public Rectangle(int width, int height) {
    this.width = width;
    this.height = height;
  }

  public int getArea() { return width * height; }
}

class Square implements Shape {
  private int side;

  public Square(int side) {
    this.side = side;
  }

  public int getArea() { return side * side; }
}
Interface Segregation Principle (ISP)

Tanım: İstemciler, kullanmadıkları arayüzlere bağımlı olmaya zorlanmamalıdır.

Neden-Sonuç İlişkisi:
  • Neden: İstemcilerin ihtiyaç duymadığı metodları implemente etme zorunluluğunu ortadan kaldırma
  • Sonuç: Daha az bağımlılık ve daha temiz arayüzler
Kötü Kullanım:
interface Animal {
  void eat();
  void sleep();
  void fly(); // Tüm hayvanlar uçamaz!
  void swim(); // Tüm hayvanlar yüzemez!
}

class Bird implements Animal {
  public void eat() { ... }
  public void sleep() { ... }
  public void fly() { ... }
  public void swim() { // Boş implementasyon - ISP ihlali! }
}
İyi Kullanım:
interface Animal {
  void eat();
  void sleep();
}

interface FlyingAnimal extends Animal {
  void fly();
}

interface SwimmingAnimal extends Animal {
  void swim();
}

class Bird implements FlyingAnimal {
  public void eat() { ... }
  public void sleep() { ... }
  public void fly() { ... }
}
Dependency Inversion Principle (DIP)

Tanım: Üst seviye modüller, alt seviye modüllere bağımlı olmamalıdır. Her ikisi de soyutlamalara bağımlı olmalıdır.

Neden-Sonuç İlişkisi:
  • Neden: Modüller arası bağımlılıkları azaltmak ve sistemi daha esnek hale getirmek
  • Sonuç: Daha kolay test edilebilirlik ve daha az bağımlılık
Kötü Kullanım:
class EmailService {
  public void sendEmail(String message) { ... }
}

class NotificationService {
  private EmailService emailService = new EmailService(); // DIP ihlali!

  public void sendNotification(String message) {
    emailService.sendEmail(message);
  }
}
İyi Kullanım:
interface MessageService {
  void sendMessage(String message);
}

class EmailService implements MessageService {
  public void sendMessage(String message) { ... }
}

class SMSService implements MessageService {
  public void sendMessage(String message) { ... }
}

@Service
class NotificationService {
  @Autowired
  private MessageService messageService; // Soyutlamaya bağımlılık

  public void sendNotification(String message) {
    messageService.sendMessage(message);
  }
}

Paket Prensipleri

Reuse Release Equivalency Principle (REP)

Tanım: Yeniden kullanılabilir birimler (paketler, modüller) sürüm kontrolü ile yönetilmeli ve açıkça tanımlanmış sürüm numaralarına sahip olmalıdır.

Neden-Sonuç İlişkisi:
  • Neden: Yeniden kullanılabilir bileşenlerin sürüm yönetimi ve bağımlılık kontrolü
  • Sonuç: Daha güvenilir ve yönetilebilir bağımlılıklar
Spring Boot'ta Uygulama:
// pom.xml - Sürüm numaraları açıkça belirtilmeli
<groupId>com.example</groupId>
<artifactId>user-service</artifactId>
<version>1.2.0</version>

// Bağımlılıklar da sürüm numaraları ile belirtilmeli
<dependencies>
  <dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-web</artifactId>
    <version>3.1.0</version>
  </dependency>
</dependencies>
Common Closure Principle (CCP)

Tanım: Aynı nedenden değişmesi muhtemel sınıflar aynı pakette toplanmalıdır.

Neden-Sonuç İlişkisi:
  • Neden: Değişikliklerin etkisini tek bir paketle sınırlandırmak
  • Sonuç: Daha az paketin yeniden derlenmesi ve dağıtılması
Spring Boot'ta Uygulama:
// Tüm kullanıcı ile ilgili sınıflar aynı pakette
com.example.user
├── User.java
├── UserController.java
├── UserService.java
├── UserRepository.java
└── UserNotFoundException.java

// Tüm ödeme ile ilgili sınıflar aynı pakette
com.example.payment
├── Payment.java
├── PaymentController.java
├── PaymentService.java
└── PaymentRepository.java
Common Reuse Principle (CRP)

Tanım: Birlikte kullanılan sınıflar aynı pakette toplanmalıdır.

Neden-Sonuç İlişkisi:
  • Neden: Paket bağımlılıklarını minimize etmek
  • Sonuç: Daha az bağımlılık ve daha temiz paket yapısı
Spring Boot'ta Uygulama:
// Security ile ilgili tüm sınıflar aynı pakette
com.example.security
├── JwtTokenUtil.java
├── JwtRequestFilter.java
├── JwtUserDetailsService.java
├── WebSecurityConfig.java
└── AuthenticationException.java
Acyclic Dependencies Principle (ADP)

Tanım: Paket bağımlılık grafiğinde döngü olmamalıdır.

Neden-Sonuç İlişkisi:
  • Neden: Döngüsel bağımlılıkların yarattığı karmaşıklığı önlemek
  • Sonuç: Daha basit ve yönetilebilir bağımlılık yapısı
Kötü Kullanım:
// Döngüsel bağımlılık - Paket A, Paket B'ye bağlı
// Paket B, Paket A'ya bağlı
com.example.packageA
└── ClassA.java // PackageB'deki ClassB'yi kullanıyor

com.example.packageB
└── ClassB.java // PackageA'daki ClassA'yı kullanıyor
İyi Kullanım:
// Ortak tipler üçüncü bir pakette tanımlanmalı
com.example.common
└── CommonTypes.java

com.example.packageA
└── ClassA.java // Common paketini kullanıyor

com.example.packageB
└── ClassB.java // Common paketini kullanıyor
Stable Dependencies Principle (SDP)

Tanım: Paketler, daha kararlı paketlere bağımlı olmalıdır.

Neden-Sonuç İlişkisi:
  • Neden: Değişken paketlerin kararlı paketleri etkilemesini önlemek
  • Sonuç: Daha istikrarlı ve güvenilir sistem
Spring Boot'ta Uygulama:
// Kararlı core paketleri
com.example.core
├── BaseEntity.java
├── BaseRepository.java
└── BaseService.java

// Daha az kararlı servis paketleri core'a bağımlı olabilir
com.example.user
├── User.java // BaseEntity'den extend eder
├── UserRepository.java // BaseRepository'den extend eder
└── UserService.java // BaseService'den extend eder
Stable Abstractions Principle (SAP)

Tanım: Kararlı paketler soyut olmalıdır, böylece kararlılıkları esnekliklerini sınırlamaz.

Neden-Sonuç İlişkisi:
  • Neden: Kararlı paketlerin genişletilebilir olmasını sağlamak
  • Sonuç: Kararlılık ve esneklik arasında denge
Spring Boot'ta Uygulama:
// Kararlı ve soyut core paketi
com.example.core
├── BaseEntity.java
├── BaseRepository.java // interface
└── BaseService.java // abstract class

// Somut implementasyonlar daha az kararlı paketlerde
com.example.user
├── UserRepositoryImpl.java // BaseRepository implementasyonu
└── UserServiceImpl.java // BaseService extendi
