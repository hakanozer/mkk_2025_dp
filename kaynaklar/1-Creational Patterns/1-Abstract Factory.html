Abstract Factory Design Pattern

Spring Boot 3.5 REST API Implementation Guide

Pattern Tanımı ve Amacı

Abstract Factory Pattern, birbiriyle ilişkili nesne ailelerini oluşturmak için arayüz sağlayan creational design pattern'dır. Bu pattern, client kodun somut sınıfları bilmeden nesne ailelerini oluşturmasına olanak tanır.

Temel Prensip: "Bir interface aracılığıyla ilgili nesneler ailesini üretmek, ancak bu nesnelerin somut sınıflarını belirtmeden."

GoF Tanımı

"Concrete sınıflarını belirtmeden, ilgili veya bağımlı nesneler ailesi oluşturmak için bir arayüz sağlar."

Mimari Yapı

AbstractFactory Interface
    ↓
ConcreteFactory1    ConcreteFactory2
    ↓                      ↓
ProductA1               ProductA2
ProductB1               ProductB2

Temel Bileşenler

AbstractFactory
İlgili ürün nesnelerini oluşturmak için operasyonlar beyan eder.
ConcreteFactory
Belirli bir ürün ailesini oluşturan operasyonları implement eder.
AbstractProduct
Bir ürün nesnesi türü için arayüz beyan eder.
ConcreteProduct
İlgili ConcreteFactory tarafından oluşturulan ürün nesnesini tanımlar.

Spring Boot 3.5 Implementation

Abstract Factory ve Product Interfaces

// Abstract Factory Interface
public interface PaymentProcessorFactory {
    PaymentProcessor createPaymentProcessor();
    PaymentValidator createPaymentValidator();
    PaymentNotifier createPaymentNotifier();
}

// Abstract Product Interfaces
public interface PaymentProcessor {
    PaymentResult processPayment(PaymentRequest request);
}

public interface PaymentValidator {
    boolean validatePayment(PaymentRequest request);
}

public interface PaymentNotifier {
    void sendNotification(PaymentResult result);
}

Concrete Factory Implementations

// Credit Card Payment Factory
@Component
@Qualifier("creditCard")
public class CreditCardPaymentFactory implements PaymentProcessorFactory {
    
    @Override
    public PaymentProcessor createPaymentProcessor() {
        return new CreditCardProcessor();
    }
    
    @Override
    public PaymentValidator createPaymentValidator() {
        return new CreditCardValidator();
    }
    
    @Override
    public PaymentNotifier createPaymentNotifier() {
        return new EmailNotifier();
    }
}

Service Layer Implementation

@Service
public class PaymentService {
    
    private final Map<String, PaymentProcessorFactory> factories;
    
    public PaymentService(Map<String, PaymentProcessorFactory> factories) {
        this.factories = factories;
    }
    
    public PaymentResult processPayment(PaymentRequest request) {
        // Factory Selection Strategy
        PaymentProcessorFactory factory = selectFactory(request.getPaymentType());
        
        // Create product family
        PaymentValidator validator = factory.createPaymentValidator();
        PaymentProcessor processor = factory.createPaymentProcessor();
        PaymentNotifier notifier = factory.createPaymentNotifier();
        
        // Business Logic
        if (!validator.validatePayment(request)) {
            throw new InvalidPaymentException("Payment validation failed");
        }
        
        PaymentResult result = processor.processPayment(request);
        notifier.sendNotification(result);
        
        return result;
    }
    
    private PaymentProcessorFactory selectFactory(PaymentType type) {
        return switch (type) {
            case CREDIT_CARD -> factories.get("creditCard");
            case DIGITAL_WALLET -> factories.get("digitalWallet");
            default -> throw new UnsupportedPaymentTypeException(type);
        };
    }
}

REST Controller

@RestController
@RequestMapping("/api/v1/payments")
@Validated
public class PaymentController {
    
    private final PaymentService paymentService;
    
    public PaymentController(PaymentService paymentService) {
        this.paymentService = paymentService;
    }
    
    @PostMapping("/process")
    public ResponseEntity<PaymentResult> processPayment(
            @Valid @RequestBody PaymentRequest request) {
        
        try {
            PaymentResult result = paymentService.processPayment(request);
            return ResponseEntity.ok(result);
        } catch (InvalidPaymentException e) {
            return ResponseEntity.badRequest().build();
        } catch (UnsupportedPaymentTypeException e) {
            return ResponseEntity.unprocessableEntity().build();
        }
    }
}

Neden-Sonuç İlişkisi

Problem: Tight Coupling ve Skalabilite

Sorun: Client kod, somut sınıflara doğrudan bağımlı olduğunda:
  • Yeni ödeme türü eklenmesi client kodda değişiklik gerektirir
  • Ürün ailesi tutarlılığı garanti edilemez
  • Test edilebilirlik azalır

Çözüm: Abstract Factory ile Decoupling

Sonuç: Abstract Factory kullanımı ile:
  • Client kod sadece interface'lere bağımlı hale gelir
  • Yeni ürün ailesi eklenmesi client kodda değişiklik gerektirmez
  • Ürün ailesi tutarlılığı factory tarafından garanti edilir

Sağladığı Faydalar

🎯 Single Responsibility
Her factory sadece kendi ürün ailesinden sorumludur. CreditCardPaymentFactory sadece kredi kartı ile ilgili nesneleri üretir.
🔒 Open/Closed Principle
Yeni ödeme türleri için extension yaparken mevcut kod değiştirilmez. Yeni factory implementation'ı yeterlidir.
🎭 Dependency Inversion
High-level modüller low-level modüllere bağımlı olmaz. PaymentService abstraction'lara bağımlıdır.
🔄 Consistency Guarantee
Factory, ürün ailesinin tutarlı olmasını garanti eder. CreditCard factory'den wallet validator çıkmaz.
🧪 Enhanced Testability
Mock factory'ler oluşturarak unit test yazılabilir. Test senaryoları için fake implementation'lar kullanılabilir.
⚡ Configuration Flexibility
Spring Boot'un dependency injection mekanizması ile runtime'da factory seçimi yapılabilir.

// Factory caching for performance
@Service
public class CachedPaymentService {
    
    private final LoadingCache<PaymentType, PaymentProcessorFactory> factoryCache;
    
    public CachedPaymentService() {
        this.factoryCache = Caffeine.newBuilder()
            .maximumSize(100)
            .expireAfterAccess(10, TimeUnit.MINUTES)
            .build(this::loadFactory);
    }
}

Error Handling Strategies

// Circuit Breaker pattern integration
@Component
public class ResilientPaymentFactory implements PaymentProcessorFactory {
    
    private final CircuitBreaker circuitBreaker;
    
    @Override
    public PaymentProcessor createPaymentProcessor() {
        return circuitBreaker.executeSupplier(() -> 
            new ResilientPaymentProcessor());
    }
}

Alternativeler ve Karşılaştırma

vs Factory Method
Abstract Factory bir ürün ailesi üretir, Factory Method tek ürün üretir. Karmaşık sistemlerde Abstract Factory tercih edilir.
vs Builder Pattern
Builder tek kompleks nesne oluşturur, Abstract Factory ilişkili nesneler ailesini oluşturur. Use case'e göre seçim yapılır.
vs Spring Configuration
@Configuration ile de factory benzeri yapılar oluşturulabilir, ancak Abstract Factory daha esnek ve runtime değişikliklere açıktır.
