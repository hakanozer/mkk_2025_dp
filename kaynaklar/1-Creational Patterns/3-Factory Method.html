Factory Method Design Pattern

Spring Boot 3.5 REST API Implementation Guide

Pattern Tanımı ve Amacı

Factory Method, bir nesnenin oluşturulmasını bir interface üzerinden yöneten, creational design pattern'dır. Bu pattern, hangi concrete class'ın instantiate edileceği kararını subclass'lara bırakarak, kodun "new" keyword'ü ile doğrudan bir nesne oluşturmasını engeller. Böylece, istemci kodun (client code) hangi concrete class'ı kullandığını bilmesine gerek kalmaz.

Temel Prensip: "Bir nesne oluşturmak için bir interface tanımla, ancak bu interface'in hangi concrete class'ı oluşturacağına subclass'lar karar versin."

GoF Tanımı

"Bir nesne oluşturmak için bir interface (method) tanımla, ancak concrete class'ın instantiate edilmesi kararını subclass'lara bırak. Factory Method, bir sınıfın (class) instantiation'ı subclass'lara delege etmesini sağlar."

Neden-Sonuç İlişkisi

Problem: Tightly Coupled Object Creation

Sorun: new keyword'ünün doğrudan kullanımı ve sıkı bağımlılıklar:
  • Rigid Codebase: Yeni bir product tipi eklemek, client code'da değişiklik yapılmasını gerektirir.
  • Violation of Open/Closed Principle: Yeni özellikler eklemek için mevcut kodu değiştirmek zorunluluğu.
  • Scattered Creation Logic: Nesne oluşturma mantığı uygulamanın farklı yerlerine yayılmıştır.
  • Testing Difficulty: Concrete dependency'ler nedeniyle unit test yazmak karmaşıklaşır.
// Anti-pattern: Tight Coupling
public class PaymentService {
    public void processPayment(String type) {
        PaymentGateway gateway;
        if ("creditCard".equals(type)) {
            gateway = new CreditCardGateway(); // Tight coupling!
        } else if ("paypal".equals(type)) {
            gateway = new PayPalGateway();
        } else {
            throw new IllegalArgumentException("Invalid gateway type");
        }
        gateway.executeTransaction();
    }
}

Çözüm: Factory Method ile Flexible Creation

Sonuç: Factory Method kullanımı ile:
  • Loose Coupling: Client code, concrete product'ı bilmek zorunda kalmaz.
  • Open/Closed Principle: Yeni bir product (örn. BitcoinGateway) eklemek, mevcut kodu değiştirmeden sadece yeni bir Concrete Factory sınıfı oluşturmayı gerektirir.
  • Centralized Logic: Nesne oluşturma mantığı tek bir yerde toplanır.
  • Enhanced Testability: Factory interface'i mock'lanarak test senaryoları kolayca oluşturulabilir.

Spring Boot 3.5 Implementation

Product ve Factory Interfaces

// Step 1: Product Interface
public interface NotificationService {
    String sendNotification(String recipient, String message);
}

// Step 2: Factory Interface
public interface NotificationFactory {
    NotificationService createNotificationService();
}

Concrete Implementations

// Step 3: Concrete Products
@Service("smsService")
public class SmsNotificationService implements NotificationService {
    @Override
    public String sendNotification(String recipient, String message) {
        return String.format("SMS sent to %s with message: '%s'", recipient, message);
    }
}

@Service("emailService")
public class EmailNotificationService implements NotificationService {
    @Override
    public String sendNotification(String recipient, String message) {
        return String.format("Email sent to %s with message: '%s'", recipient, message);
    }
}

// Step 4: Concrete Factories
@Component("smsFactory")
public class SmsNotificationFactory implements NotificationFactory {
    @Override
    public NotificationService createNotificationService() {
        return new SmsNotificationService();
    }
}

@Component("emailFactory")
public class EmailNotificationFactory implements NotificationFactory {
    @Override
    public NotificationService createNotificationService() {
        return new EmailNotificationService();
    }
}

Service Layer: Factory Injection

@Service
public class NotificationServiceManager {
    
    private final Map<String, NotificationFactory> factories;

    // Spring automatically injects all beans of type NotificationFactory into a Map
    public NotificationServiceManager(Map<String, NotificationFactory> factories) {
        this.factories = factories;
    }
    
    public String sendNotificationByType(String type, String recipient, String message) {
        NotificationFactory factory = factories.get(type + "Factory");
        if (factory == null) {
            throw new IllegalArgumentException("Invalid notification type: " + type);
        }
        NotificationService service = factory.createNotificationService();
        return service.sendNotification(recipient, message);
    }
}

REST Controller: Loose Coupling

@RestController
@RequestMapping("/api/v1/notifications")
public class NotificationController {
    
    private final NotificationServiceManager serviceManager;

    public NotificationController(NotificationServiceManager serviceManager) {
        this.serviceManager = serviceManager;
    }
    
    @PostMapping("/send")
    public ResponseEntity<String> sendNotification(
            @RequestParam String type,
            @RequestParam String recipient,
            @RequestParam String message) {
        
        try {
            String result = serviceManager.sendNotificationByType(type, recipient, message);
            return ResponseEntity.ok(result);
        } catch (IllegalArgumentException e) {
            return ResponseEntity.badRequest().body(e.getMessage());
        }
    }
}

İleri Seviye Factory Method Uygulamaları

Strategy Pattern ile Entegrasyon

Factory Method genellikle Strategy Pattern ile birlikte kullanılır. Factory, doğru Strategy nesnesini (örneğimizde NotificationService) oluşturmaktan sorumluyken, istemci (NotificationServiceManager) bu nesneyi kullanarak bir işlemi gerçekleştirir. Bu, sorumlulukların net bir şekilde ayrılmasını sağlar.

Spring'in Map Injection Özelliği

// Spring'in Map-based injection'ı, Factory Method'ın en modern implementasyonudur.
// notificationFactoryMap key olarak bean adını, value olarak ise bean instance'ını tutar.
// Bu sayede switch-case veya if-else bloklarına ihtiyaç kalmaz.
@Service
public class NotificationFactoryLocator {

    private final Map<String, NotificationFactory> notificationFactoryMap;

    public NotificationFactoryLocator(Map<String, NotificationFactory> notificationFactoryMap) {
        this.notificationFactoryMap = notificationFactoryMap;
    }

    public NotificationFactory getFactory(String type) {
        return notificationFactoryMap.get(type + "Factory");
    }
}

Sağladığı Faydalar

🔌 Loose Coupling (Gevşek Bağlılık)
İstemci kodu, concrete class'lara bağımlı değildir. Sadece product interface'i ile çalışır, bu da esnekliği artırır.
🔄 Enhanced Extensibility (Genişletilebilirlik)
Yeni bir product tipi eklemek, mevcut kodu değiştirmeden (Open/Closed Prensibi) kolayca yapılır. Yeni bir factory ve product sınıfı eklemek yeterlidir.
🧪 Better Testability (Daha İyi Test Edilebilirlik)
Factory method'lar, testler sırasında mock'lanabilir. Bu, product'ın karmaşık bağımlılıklarından arındırılmış unit testler yazmayı mümkün kılar.
✅ Single Responsibility Principle
Nesne oluşturma mantığı, asıl iş mantığından ayrılır. Her sınıf kendi sorumluluğunu yerine getirir.

Alternativeler ve Karşılaştırma

Pattern Use Case Complexity Factory Method ile Farkı
Abstract Factory İlişkili nesne aileleri Yüksek Tek bir nesne yerine, bir nesne ailesini oluşturur.
Builder Pattern Karmaşık, adım adım nesne oluşturma Orta Tek bir "build" method'u ile karmaşık nesneleri inşa eder.
Singleton Uygulama genelinde tek bir nesne Düşük Her zaman aynı nesne instance'ını döndürür.
Plain "new" Basit, bağımlılıksız nesneler Çok Düşük Sıkı bağımlılık yaratır, esnek değildir.

✅ Factory Method Avantajları

  • Esneklik: Yeni product'lar kolayca eklenebilir.
  • Daha Az Bağlılık: Client, concrete class'ları bilmek zorunda kalmaz.
  • Kod Organizasyonu: Creation logic'i tek bir yerde toplanır.
  • Çeviklik: Değişiklikler için daha az etki alanı (ripple effect).

⚠️ Dikkat Edilmesi Gerekenler

  • Artan Sınıf Sayısı: Her yeni product için yeni bir factory sınıfı gerekebilir.
  • Ekstra Karmaşıklık: Basit nesneler için overkill olabilir.
  • Factory "Hell": Çok fazla factory sınıfı yönetimi zorlaştırabilir.
  • Kod Düzeni: Doğru şekilde uygulanmazsa karmaşa yaratır.

Production Ready Implementation

// Dinamik olarak factory seçimi ve kullanım örneği
@Service
public class ProductionNotificationService {

    private final ApplicationContext context;

    public ProductionNotificationService(ApplicationContext context) {
        this.context = context;
    }

    public NotificationResponse processAndSend(NotificationRequest request) {
        // Dynamic factory lookup from Spring Context
        NotificationFactory factory = context.getBean(request.getType() + "Factory", NotificationFactory.class);
        NotificationService service = factory.createNotificationService();

        String result = service.sendNotification(request.getRecipient(), request.getMessage());
        
        return new NotificationResponse(true, result);
    }
}
Conclusion: Factory Method, özellikle bir interface'in farklı implementasyonlarına ihtiyaç duyulan ve bu implementasyonların dinamik olarak seçilip oluşturulması gereken senaryolarda kritik bir öneme sahiptir. Spring'in güçlü IoC (Inversion of Control) container'ı ile birleştiğinde, "new" keyword'ünün neden olduğu katı bağımlılıklardan kurtularak çok daha esnek, genişletilebilir ve sürdürülebilir bir kod yapısı oluşturulabilir.
